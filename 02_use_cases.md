# Use Case 1 (см. файл пункт 4.3.1)
**Название**: Импорт расписания через Excel-файл.

**Роль**: Староста.

**Предусловие**:
1. Создана база данных.
2. Староста зарегистрирован в боте (вошел как староста отпределенной группы).

**Основной поток**:
1. Староста вводит команду `/import_schedule` для импорта расписания в базу данных.
2.  Система выводит три способа (Excel-файл, фото, ввести вручную) загрузки расписания.
3. Староста выбирает для загрузки данных Excel-файл. 
4. Система предлагает скачать шаблон для заполнения расписания.
5. Староста вводит команду `/template`.
6. Система отправляет Excel-файл `schedule_template.xlsx`.
7. Староста скачивает, заполняет файл и отправляет его обратно в систему.
8. Система выводит распознанные данные пользователю для проверки и запрашивает подтверждение (команды `/confirm_schedule` или `/edit_schedule`).
9. Староста, если все верно, вводит команду `/confirm_schedule`.
10. Система сохраняет данные и оповещает об этом пользователей.

**Альтернативный поток**:
1. Система вывела в 8 пункте основного потока данные с ошибками.  
1.1 Староста вводит команду `/edit_schedule`.  
1.2 Система отправляет пользователю файл (с ошибками).  
1.3 Староста скачивает себе файл, исправляет ошибки, загружает исправленный файл в систему.  
   
**Постусловие**:
1. В системе загружено расписание.
2. Группа получила уведомление о загруженном расписании. 

***В этом Use Case под системой подразумевается бот NAKOL.***

# Use Case 2 (см. файл пункт 4.3.2)
**Название**: Импорт расписания через фото.

**Роль**: Староста.

**Предусловие**:
1. Создана база данных.
2. Староста зарегистрирован и находится в диалоге с ботом.

**Основной поток**:
1. Староста вводит команду `/import_schedule` для импорта расписания в базу данных.
2. Система выводит три способа загрузки расписания (Excel-файл, фото, ввести вручную).
3. Староста выбирает для загрузки данных фото.
4. Система запрашивает у пользователя фото расписания и предупреждает, что оно должно быть чётким.
5. Староста отправляет фото в систему.
6. Система уведомляет пользователя о начале процесса распознавания текста (OCR).
7. Система обрабатывает фото, распознаёт текст и структурирует данные (день недели, время, предмет, аудитория).
8. Система выводит распознанные данные пользователю для проверки и запрашивает подтверждение (команды `/confirm_schedule` или `/edit_schedule`).
9. Староста, если все данные верны, вводит команду `/confirm_schedule`.
10. Система сохраняет данные в базе и оповещает участников группы о загруженном расписании.

**Альтернативный поток**:
1. На шаге 8 основного потока система вывела данные с ошибками, либо пользователь хочет их исправить.  
1.1 Староста вводит команду `/edit_schedule`.  
1.2 Система повторно запрашивает фото расписания (возврат к шагу 4 основного потока).  
1.3 Староста отправляет новое (или то же, более чёткое) фото.  
1.4 Поток продолжается с шага 6 основного потока (повторная обработка).  
   
**Постусловие**:
1. В системе загружено и сохранено расписание для группы.
2. Участники группы получили уведомление о загруженном расписании.

***В этом Use Case под системой подразумевается бот NAKOL.***

# Use Case 3 (см. файл пункт 4.3.3)

**Название**: Импорт расписания через ручной ввод.

**Роль**: Староста.

**Предусловие**:
1. Создана база данных.
2. Староста зарегистрирован и находится в диалоге с ботом.

**Основной поток**:
1.  Староста вводит команду `/import_schedule` для импорта расписания в базу данных.
2.  Система выводит три способа загрузки расписания (Excel-файл, фото, ввести вручную).
3.  Староста выбирает ручной ввод данных.
4.  Система предлагает вводить занятия по одному, указывает формат: «День, Время начала, Время конца, Предмет» (например: пн, 10:00, 11:30, Математика) и сообщает, что для завершения нужно отправить `/done`.
5.  Староста вводит данные первого занятия в указанном формате.
6.  Система проверяет корректность формата и сохраняет занятие, после чего выводит подтверждение: «Добавлено: [Предмет] ([день] [время])» и повторяет приглашение ввести следующее или `/done`.
7.  Шаги 5–6 повторяются для всех занятий, которые хочет добавить староста.
8.  Староста вводит команду `/done`.
9.  Система сохраняет все введённые данные в базе и оповещает участников группы о загруженном расписании.

**Альтернативный поток**:
1. На шаге 5 или 7 основного потока староста вводит данные в неверном формате (например, пропущена запятая, неверный день недели, некорректное время).  
1.1 Система определяет ошибку и выводит сообщение: «Неверный формат. Попробуй ещё раз.».  
1.2 Система остаётся в состоянии ожидания ввода того же занятия (возврат к шагу 5 для текущего занятия, без перехода к следующему).  

**Постусловие**:
1.  В системе загружено и сохранено расписание для группы.
2.  Участники группы получили уведомление о загруженном расписании.

***В этом Use Case под системой подразумевается бот NAKOL.***

# Use Case 4 (см. файл пункт 4.4.1)
**Название**: Импорт домашних заданий через фото.

**Роль**: Староста.

**Предусловие**:
1. Создана база данных группы.
2. Староста авторизован и является администратором группы.

**Основной поток**:
1. Староста вводит команду `/import_homework` для загрузки домашних заданий.
2. Система предлагает выбрать способ загрузки: фото (OCR) или ввод вручную.
3. Староста выбирает пункт «Отправить фото».
4. Система запрашивает отправку фото, напоминая, что текст должен быть разборчивым.
5. Староста отправляет фото с перечнем домашних заданий.
6. Система выполняет распознавание текста (OCR), выделяет отдельные задания и пытается определить для каждого дедлайн (если указан).
7. Система выводит список найденных заданий, помечая те, у которых дедлайн не обнаружен, и предлагает два варианта:
`/add_dates` – добавить даты вручную; `/send_to_group` – отправить группе как есть.
8. Староста выбирает `/send_to_group`.
9. Система рассылает задания всем участникам группы (с возможностью отслеживать выполнение) и уведомляет старосту об успешной отправке.

**Альтернативный поток**:
1. На шаге 8 староста выбирает `/add_dates`.  
1.1 Система поочерёдно запрашивает дату для каждого задания, у которого дедлайн не указан (например, в формате ДД.ММ).  
1.2 Староста вводит дату или пропускает, оставляя без дедлайна.  
1.3 После обработки всех заданий система возвращается к шагу 7 или сразу выполняет отправку.  

2. На шаге 6 системе не удаётся распознать текст (фото слишком тёмное, нечёткое, нет текста, текст не распознан).  
2.1 Система выводит сообщение об ошибке и предлагает отправить другое фото или выбрать другой способ загрузки.    
2.2 Староста выбирает "Отправить фото повторно" или "Ввести данные вручную".  
2.3 Возврат к шагу 4 или к шагу 2 основного потока соотвественно пункту 2.2 альтернативного потока.    
   
**Постусловие**:
1. Домашние задания отправлены всем участникам группы.
2. Участники могут отмечать выполнение заданий, система отслеживает прогресс.
3. В журнале группы фиксируется факт отправки заданий.
