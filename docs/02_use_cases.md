# Use Case 1 (см. файл "NAKOL основное" пункт 4.3.1)
**Название**: Импорт расписания через Excel-файл.

**Роль**: 
1. Староста.
2. Индивидуал.
3. Система.

**Предусловие**:
1. Создана база данных.
2. Студент зарегистрирован в боте.

**Основной поток**:
1. Студент вводит команду `/import_schedule` для импорта расписания в базу данных.
2. Система выводит три способа (Excel-файл, фото, ввести вручную) загрузки расписания.
3. Студент выбирает для загрузки данных Excel-файл. 
4. Система предлагает скачать шаблон для заполнения расписания.
5. Студент вводит команду `/template` - запрашивает шаблон.
6. Система отправляет Excel-файл `schedule_template.xlsx`.
7. Студент скачивает, заполняет файл и отправляет его обратно в систему.
8. Система выводит распознанные данные пользователю для проверки и запрашивает подтверждение (команды `/confirm_schedule` или `/edit_schedule`).
9. Студент, если все верно, вводит команду `/confirm_schedule`.
10. Система сохраняет данные и оповещает об этом пользователей.

**Альтернативный поток**:
1. Система вывела в 8 пункте основного потока данные с ошибками.  
1.1 Студент вводит команду `/edit_schedule`.  
1.2 Система отправляет пользователю файл (с ошибками).  
1.3 Студент скачивает себе файл, исправляет ошибки, загружает исправленный файл в систему.  
   
**Постусловие**:
2. В системе загружено расписание.
3. Группа получила уведомление о загруженном расписании. 

***В этом Use Case под системой подразумевается бот NAKOL.***  
***В этом Use Case под студентом подразумевается староста или индивидуал.***

# Use Case 2 (см. файл "NAKOL основное" пункт 4.3.2)
**Название**: Импорт расписания через фото.

**Роль**: 
1. Староста.
2. Индивидуал.
3. Система.

**Предусловие**:
1. Создана база данных.
2. Студент зарегистрирован и находится в диалоге с ботом.

**Основной поток**:
1. Студент вводит команду `/import_schedule` для импорта расписания в базу данных.
2. Система выводит три способа загрузки расписания (Excel-файл, фото, ввести вручную).
3. Студент выбирает для загрузки данных фото.
4. Система запрашивает у пользователя фото расписания и предупреждает, что оно должно быть чётким.
5. Студент отправляет фото в систему.
6. Система обрабатывает фото, распознаёт текст и структурирует данные (день недели, время, предмет, аудитория).
7. Система выводит распознанные данные пользователю для проверки и запрашивает подтверждение (команды `/confirm_schedule` или `/edit_schedule`).
8. Студент, если все данные верны, вводит команду `/confirm_schedule`.
9. Система сохраняет данные в базе и оповещает участников группы о загруженном расписании.

**Альтернативный поток**:
1. На шаге 7 основного потока система вывела данные с ошибками, либо пользователь хочет их исправить.  
1.1 Студент вводит команду `/edit_schedule`.  
1.2 Система повторно запрашивает фото расписания (возврат к шагу 4 основного потока).  
1.3 Студент отправляет новое (или то же, более чёткое) фото.  
1.4 Поток продолжается с шага 6 основного потока (повторная обработка).  
   
**Постусловие**:
2. В системе загружено и сохранено расписание для группы.
3. Участники группы получили уведомление о загруженном расписании.

***В этом Use Case под системой подразумевается бот NAKOL.***  
***В этом Use Case под студентом подразумевается староста или индивидуал.***

# Use Case 3 (см. файл "NAKOL основное" пункт 4.3.3)

**Название**: Импорт расписания через ручной ввод.

**Роль**: 
1. Староста.
2. Индивидуал.
3. Система.

**Предусловие**:
1. Создана база данных.
2. Студент зарегистрирован и находится в диалоге с ботом.

**Основной поток**:
1.  Студент вводит команду `/import_schedule` для импорта расписания в базу данных.
2.  Система выводит три способа загрузки расписания (Excel-файл, фото, ввести вручную).
3.  Студент выбирает ручной ввод данных.
4.  Система предлагает вводить занятия по одному, указывает формат: `«День, Время начала, Время конца, Предмет»` (например: пн, 10:00, 11:30, Математика) и сообщает, что для завершения нужно отправить `/done`.
5.  Студент вводит данные первого занятия в указанном формате.
6.  Система проверяет корректность формата и сохраняет занятие, после чего выводит подтверждение: «Добавлено: [Предмет] ([день] [время])» и повторяет приглашение ввести следующее или `/done`.
7.  Шаги 5–6 повторяются для всех занятий, которые хочет добавить Студент.
8.  Студент вводит команду `/done`.
9.  Система сохраняет все введённые данные в базе и оповещает участников группы о загруженном расписании.

**Альтернативный поток**:
1. На шаге 5 или 7 основного потока Студент вводит данные в неверном формате (например, пропущена запятая, неверный день недели, некорректное время).  
1.1 Система определяет ошибку и выводит сообщение: `«Неверный формат. Попробуй ещё раз.»`.  
1.2 Система остаётся в состоянии ожидания ввода того же занятия (возврат к шагу 5 для текущего занятия, без перехода к следующему).  

**Постусловие**:
1.  В системе загружено и сохранено расписание для группы.
2.  Участники группы получили уведомление о загруженном расписании.

***В этом Use Case под системой подразумевается бот NAKOL.***  
***В этом Use Case под студентом подразумевается староста или индивидуал.***

# Use Case 4 (см. файл "NAKOL основное" пункт 4.4.1)
**Название**: Импорт домашних заданий через фото.

**Роль**: 
1. Староста.
2. Индивидуал.
3. Система.

**Предусловие**:
1. Создана база данных группы.
2. Студент авторизован.

**Основной поток**:
1. Студент вводит команду `/import_homework` для загрузки домашних заданий.
2. Система предлагает выбрать способ загрузки: фото (OCR) или ввод вручную.
3. Студент выбирает пункт «Отправить фото».
4. Система запрашивает отправку фото.
5. Студент отправляет фото с перечнем домашних заданий.
6. Система загружает фото в базу данных.
7. Система выводит список заданий, помечая те, у которых дедлайн не обнаружен, и предлагает два варианта:
`/add_dates` – добавить даты вручную; `/send_to_group` – отправить группе как есть.
8. Студент выбирает `/send_to_group`.
9. Система рассылает задания всем участникам группы (с возможностью отслеживать выполнение) и уведомляет старосту об успешной отправке.

**Альтернативный поток**:
1. На шаге 8 Студент выбирает `/add_dates`.  
1.1 Система поочерёдно запрашивает дату для каждого задания, у которого дедлайн не указан (в формате ДД.ММ.ГГГГ).  
1.2 Студент вводит дату или пропускает, оставляя без дедлайна.  
1.3 После обработки всех заданий система возвращается к шагу 7 или сразу выполняет отправку.  

2. На шаге 6 системе не удаётся загрузить фото.  
2.1 Система выводит сообщение об ошибке и предлагает отправить другое фото или выбрать другой способ загрузки.    
2.2 Студент выбирает "Отправить фото повторно" или "Ввести данные вручную".  
2.3 Возврат к шагу 4 или к шагу 2 основного потока соотвественно пункту 2.2 альтернативного потока.    
   
**Постусловие**:
1. Домашние задания отправлены всем участникам группы.
2. Участники могут отмечать выполнение заданий, система отслеживает прогресс.
3. В журнале группы фиксируется факт отправки заданий.

***В этом Use Case под системой подразумевается бот NAKOL.***  
***В этом Use Case под студентом подразумевается староста или индивидуал.***

# Use Case 5 (см. файл "NAKOL основное" пункт 4.4.2)

**Название**: Импорт домашних заданий через ручной ввод.

**Роль**: 
1. Староста.
2. Индивидуал. 
3. Система.

**Предусловие**:
1. Создана база данных группы.
2. Студент авторизован.

**Основной поток**:
1.  Студент вводит команду `/import_homework` для загрузки домашних заданий.
2.  Система предлагает выбрать способ загрузки: фото (OCR) или ввод вручную.
3.  Студент выбирает пункт «Ввести вручную».
4.  Система запрашивает входные данные: «Предмет, Задание, Дедлайн» и сообщает, что для завершения нужно отправить `/done`.
5.  Студент вводит данные первого задания в указанном формате.
6.  Система проверяет корректность формата и сохраняет задание, после чего выводит подтверждение: `«Добавлено: [Предмет] [Название] (до [Дедлайн])»` и предлагает ввести следующее или `/done`.
7.  Шаги 5–6 повторяются для всех заданий, которые хочет добавить Студент.
8.  Студент вводит команду `/done`.
9.  Система выводит итоговое количество добавленных заданий и спрашивает: `«Отправить группе?»`.
10. Если все верно, Студент вводит команду `/send_to_group`.
11. Система рассылает задания всем участникам группы и уведомляет старосту об успешной отправке.

**Альтернативный поток**:
1. На шаге 5 или 7 основного потока Студент вводит данные в неверном формате (например, не хватает запятых, неверный формат даты).  
1.1 Система определяет ошибку и выводит сообщение: `«Неверный формат. Попробуй ещё раз. Пример: Математика, №345, 05.03.2026»`.  
1.2 Система остаётся в состоянии ожидания ввода того же задания - 5 шаг основного потока.  

2. Пользователь не хочет указывать дедлайн для задания.  
2.1 Студент вводит: `Предмет, Название, ` (оставляет поле дедлайна пустым, но с запятой).  
2.2 Система интерпретирует это как задание без срока, сохраняет его и выводит: `«Добавлено: [Предмет] [Название] (дедлайн не указан)»`.  
2.3 Поток продолжается с шага 7.  

**Постусловие**:
1.  Домашние задания отправлены всем участникам группы.
2.  Участники могут отмечать выполнение заданий, система отслеживает прогресс.
3.  В журнале группы фиксируется факт отправки заданий.

***В этом Use Case под системой подразумевается бот NAKOL.***  
***В этом Use Case под студентом подразумевается староста или индивидуал.***

# Use Case 6 
**Название**: Отметка задания выполненным.

**Роль**:
1. Студент.
2. Система.

**Предусловие**:
1. Студент авторизован.

2. На 3 шаге **основного потока** студент введет неверное и несуществующее `ID`, система выведет сообщение об ошибке и запросит повтор сообщения.

**Постусловие**:
1. Система пометила задачу как выполненную, изменила прогресс.

   1.2 Система повторно запрашивает фото расписания (возврат к шагу 4 основного потока).   
   1.3 Студент отправляет новое (или то же, более чёткое) фото.  
   1.4 Поток продолжается с шага 6 основного потока (повторная обработка).  
   
**Постусловие**:
2. В системе загружено и сохранено расписание сессии.  
3. Участники группы получили уведомление о загруженном расписании сессии.

***В этом Use Case под системой подразумевается бот NAKOL.***  
***В этом Use Case под студентом подразумевается староста или индивидуал.***

# Use Case 12
**Название**: Импорт расписания через ручной ввод.

**Роль**: 
1. Староста.
2. Индивидуал.
3. Система.

**Предусловие**:
1. Создана база данных.
2. Студент зарегистрирован.

**Основной поток**:
1.  Студент вводит команду `/import_session` для импорта расписания сессии в базу данных.
2.  Система выводит три способа загрузки расписания (Excel-файл, фото (OCR), ввести вручную).
3.  Студент выбирает ручной ввод данных.
4.  Система предлагает вводить экзамены по одному, указывает формат: `«День, Время начала, Время конца, Предмет»` (например: пн, 10:00, 11:30, Математика) и сообщает, что для завершения нужно отправить `/done`.
5.  Студент вводит данные первого экзамена в указанном формате.
6.  Система проверяет корректность формата и сохраняет занятие, после чего выводит подтверждение: «Добавлено: [Предмет] ([день] [время])» и повторяет приглашение ввести следующее или `/done`.
7.  Шаги 5–6 повторяются для всех экзаменов, которые хочет добавить Студент.
8.  Студент вводит команду `/done`.
9.  Система сохраняет все введённые данные.

**Альтернативный поток**:
1. На шаге 5 или 7 основного потока Студент вводит данные в неверном формате (например, пропущена запятая, неверный день недели, некорректное время).  
1.1 Система определяет ошибку и выводит сообщение: `«Неверный формат. Попробуй ещё раз.»`.  
1.2 Система остаётся в состоянии ожидания ввода того же экзамена (возврат к шагу 5 для текущего экзамена, без перехода к следующему).  

**Постусловие**:
1.  В системе загружено и сохранено расписание.
2.  Участники группы получили уведомление о загруженном расписании сессии.

***В этом Use Case под системой подразумевается бот NAKOL.***  
***В этом Use Case под студентом подразумевается староста или индивидуал.***
